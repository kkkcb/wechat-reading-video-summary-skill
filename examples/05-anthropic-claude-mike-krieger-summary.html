01

AI 时代的效率幻象

Burbn → Instagram 的启示

Mike Krieger 回忆 Instagram 的诞生：他们最初做 Burbn 花了整整一年，功能繁杂，签到、计划、照片分享应有尽有——但行不通。后来痛下决心转型，只保留照片分享核心功能，Instagram 由此诞生，而这次转型只花了 3 个月。

令人震撼的是，Mike 用 Claude 仅花 2 小时就重建了当年 Burbn 的全部功能。更令人惊讶的是，Claude 甚至预知了产品的演进方向——它提前在代码中加入了滤镜功能，仿佛"知道" Instagram 最终会走向何方。

模型非常擅长"增加功能"，但它不擅长"知道该砍掉什么"。这种判断力需要在现实世界中大量实践才能理解。 —— Mike Krieger

AI 加速开发 ≠ 产品成功

核心矛盾：AI 可以让你在几小时内构建出任何东西，但它无法告诉你应该构建什么。模型的能力在于"增加"，而非"精简"——这正是产品成功的关键分水岭。

Mike 反思：为何在 AI 加速开发的时代，依然没有出现很多现象级产品？原因在于：即使 AI 能极大提高开发效率，但"你想对世界做出什么样的干预"——这个根本性问题，仍然需要时间磨练、试错和深入思考。

⚡关键洞察

效率的提升并没有降低产品成功的门槛。相反，它让"快速试错"变得更容易的同时，也让"功能堆砌"变得更具诱惑力。真正的产品能力，在于知道什么该保留、什么该砍掉。

Vibe Coding（氛围感编程）

定义：开发者不再手写具体的代码行，而是通过"传达意图、调整方向、描述感觉"来驱动 AI 生成整个应用。编程的范式从"精确指令"转变为"意图描述"。

直觉：就像你不是亲手盖房子，而是告诉建筑师"我要一个温暖的、有落地窗的房子"，建筑师就帮你盖好了。你关注的是"感觉"，而非每一块砖的位置。

原理：开发者用自然语言描述需求 → AI（如 Claude Code）理解意图 → 生成完整代码 → 开发者审查和调整。整个过程，开发者扮演的是"导演"角色，AI 是"执行团队"。

对比：传统编程是精确指令——"建一堵 3 米高的墙"；Vibe Coding 是意图驱动——"让这个空间感觉更开放"。前者控制每一个细节，后者关注整体方向。

代价：效率极高但容易失控——功能越堆越多，产品越来越复杂，开发者逐渐失去了对"什么该留、什么该砍"的直觉判断。正如 Mike Krieger 所警告的，AI 最擅长的恰恰是"加功能"，而这恰恰是最危险的。

02

温室树木的隐喻

室内树 vs 野生树

Dan Shipper 的隐喻：如果你在室内种一棵树，不让它接触风，它就不会长得那么强壮。它需要外力的磨练才能长成真正的树。室内树看似完整——枝繁叶茂、绿意盎然——但它的根系浅薄、枝干纤细，和经历风雨的野外树完全不同。

AI 开发就像在室内种一棵完整的树——你迅速拥有了这个完整的东西，但它每一步都没有经过直觉的锤炼和经验的磨练。每一个功能的添加，都不是因为"用户需要"，而是因为"AI 可以做到"。

图 1：室内树 vs 野生树 —— AI 生成的产品（左）枝繁叶茂但根基浅薄，经历真实世界磨练的产品（右）枝干精炼但根基深厚

功能冗余的陷阱

Mike 在实验室做项目时，亲身体会到了这种诱惑。在进入 Early Access 之前，他就把 V1 版本过度开发了。他的内心独白是："既然有这个选项，为什么不把那个也加上？"——这正是 Vibe Coding 带来的功能膨胀陷阱。

电视剧角色隐喻

向某人展示一棵完全长成的树，就像"你被直接扔进电视剧最后一集"——"等等，这些人都是谁？我居然被要求已经掌握了所有背景信息。" —— Dan Shipper

Dan 自己的经历更加印证了这一点。他用 Vibe Coding 做应用 Proof 时，因为太好玩、太上瘾，创造出了一个难用的庞然大物。后来他受到 Monologue（一个极简语音转文字应用）的启发，毅然扔掉原来的产品，重做了一个极简版本——结果在 Every 内部病毒式传播。

🌱核心教训

室内树看似完整，但缺乏经风雨锤炼的根基。用 AI 快速构建的产品，如果每一项功能都不是为了解决真实的用户问题而存在，那它就是一棵温室里的树——看起来枝繁叶茂，一阵风就可能吹倒。

03

推倒重写的勇气

重写不再是禁区

经典理论：Fred Brooks 在《人月神话》中严厉警告不要重写软件，还有著名的"第二系统综合症"——设计师在设计第二个系统时，倾向于把第一个系统中省略的所有"精彩想法"一股脑加上，导致过度设计。此外，Netscape 因重写浏览器而拖垮公司的教训，至今仍被引用。

Mike 的不同看法：这些理论依然有道理，但时代已经变了。现在模型可以帮你做 Diff（代码对比），检查是否遗漏了第一版本的核心内容。重写不再是拖垮公司长达一年的计划——现在只需要几天。

《人月神话》与第二系统综合症

定义：Fred Brooks 1975 年出版的软件工程经典著作，提出了著名的"布鲁克斯法则"——向一个延期的软件项目增加人手只会使它更加延期。

直觉：就像厨房里已经有太多厨师，再加人只会让场面更混乱，而不是更高效。沟通成本随人数呈指数级增长。

原理：第二系统综合症（Second-System Effect）指设计师在设计第二个系统时，倾向于加入第一个系统中刻意省略的所有"精彩想法"，导致过度设计。因为第一次克制了自己的创意冲动，第二次就变本加厉地释放。

对比：AI 时代，重写的成本从"一年"缩短到"几天"。模型能做 Diff 对比，大幅降低了遗漏风险。这让"第二系统综合症"的危险大大降低——你不需要一次性加完所有功能，因为重写的代价很低。

代价：仍需警惕——即使重写成本低，频繁推倒重来也可能浪费团队信心和用户信任。关键是在"迭代改进"和"推倒重来"之间找到正确节奏。

更早发布的哲学

Anthropic 的实际做法：多个项目先构建完整版本，发现搞复杂了或核心假设错误后，直接推倒做 V2。团队并不觉得"白建了一年"，而是"那是上周的事，这周我可以重做"。

不能假设在发布前就知道产品需要每一个功能。你必须更早地发布，更早地获取反馈。 —— Mike Krieger

Antfooding 内部测试

Anthropic 有一个内部试用者群体叫 "Antfooders"——源自"吃自己的狗粮"（Dogfooding）。Anthropic 员工自称 Ants（蚂蚁），所以 Dogfooding 变成了 Antfooding。Co-worker 产品就是典型案例：打磨了很久，但一旦决定发布，就以最简约方式在 10 天内推出 V1。

Dogfooding 与 Antfooding

定义：Dogfooding = "吃自己的狗粮"（Eat your own dog food），即公司内部使用自己开发的产品。Anthropic 版本叫 Antfooding——Ants（蚂蚁，Anthropic 员工自称）+ Dogfooding 的合成词。

直觉：就像厨师自己做菜自己先尝，确认味道没问题再端给客人。如果连自己都不愿意用，怎么指望用户喜欢？

原理：内部测试能发现大量 Bug 和体验问题，因为团队成员对产品的容错度反而更低。但也有局限性——内部用户对产品太熟悉，无法代表真实用户的首次体验。

对比：Antfooding 只是第一步，真实用户反馈才是最终验证。Mike 坦言："在这产品真正进入现实世界之前，内部测试也只能带你走这么远。"但至少，Antfooding 能确保基本质量。

代价：过早发布可能伤害品牌，过晚发布可能错过市场窗口。Anthropic 的"10 天法则"是一个平衡点——足够短以保持敏捷，足够长以确保基本质量。

智能体原生产品设计

什么是智能体原生

Dan Shipper 的定义：智能体必须能够像用户一样使用应用中的任何功能。不是通过 API 间接访问，而是像真人一样——点击按钮、填写表单、浏览菜单。

Mike Krieger 的三层解读：

  • 1
    第一层：普通人终于能"用电脑"了
    以前只有懂命令行的人才能做到的事，现在 Claude 可以帮你做。AI 降低了使用计算能力的门槛。
  • 2
    第二层：Claude AI 仍需进化
    Claude Code 已经做得很出色，但 Claude AI（网页版/App）仍有差距。用户想让 Claude 把作品添加到项目知识库，Claude 回答"让我告诉你去哪里点按钮"——这是错的，它应该原生完成操作。
  • 3
    第三层：智能体可以修改"运行框架"本身
    不需要坐在那想"我希望 Gmail 的运作方式能稍微改一下"，而是直接要求智能体去改。这不是使用工具，而是重塑工具。

Claude Code 的启示

Mike 讲述了一个精彩的故事：他让 Claude Code 把 Dan 关于"智能体原生"的文章打包成一个 Skill。Claude 首先查找自己的"技能开发技能"（meta-skill），然后按照这个技能的标准流程，创建并安装了新技能——整个过程展示了 AI 掌握自身知识后解锁的巨大能力。

验证（Verification）的新范式：传统产品可以写端到端的逻辑测试，覆盖每条用户路径。但智能体原生产品的魅力恰恰在于不可预测性——AI 能发现开发者从未想过的使用方式。实验室反复讨论的核心问题是：如何提高验证的保真度（Fidelity），在保持灵活性的同时确保安全。

软件的未来形态

智能体原生设计代表了软件设计哲学的根本转变。传统设计追求护栏（Guardrails）和确定性，确保用户只能走预设路径。AI 时代的产品需要创造超灵活的可扩展体，让智能体能自主调用底层功能、理解用户意图——但这同时带来了验证和安全性的巨大挑战。

智能体原生（Agent-Native）设计

定义：软件从一开始就设计为可以让 AI 智能体像人类用户一样使用所有功能，而非仅仅提供 API 接口。智能体不是"旁观者"，而是"第一公民"。

直觉：传统软件像自动售货机——你只能按已有的按钮。智能体原生软件像有万能钥匙的管家——可以打开任何门、使用任何工具，甚至改造房屋结构。

原理：传统设计理念追求护栏（Guardrails）和测试，确保用户只能走一条路径。智能体原生设计创造超灵活的可扩展体，AI 可以自主调用底层功能、理解用户意图，甚至发现开发者未设想的使用方式。

对比：传统工程师追求确定性 → 用户只能走预设路径。AI 时代产品需要灵活性 → 智能体能探索任何路径。从"高速公路"到"开放丛林"。

代价：灵活性增加导致测试难度暴增。如何在保持灵活性的同时确保可靠性和安全性，是当前 AI 产品面临的核心挑战。验证保真度（Verification Fidelity）成为新的技术前沿。

运行框架（Harness）

定义：围绕 AI 模型构建的整套基础设施。模型负责推理（思考），Harness 负责执行（动手）。两者结合，才构成一个完整的 AI 系统。

直觉：就像赛车——模型是车手（做决策），Harness 是整辆赛车（引擎、轮胎、方向盘）。AI 时代的突破是：车手现在可以改装赛车本身。

原理：Harness 包括文件系统访问、Shell 执行、API 调用、工具集成等。在 Claude Code 中，Harness 就是终端环境 + 文件读写 + 命令执行能力的总和。

对比：固定 Harness = 工具不可改变，用户只能使用预设功能。可修改 Harness = 智能体可以改变自己的工具集和工作方式，根据任务需求动态适配。

代价：让 AI 修改自己的执行环境增加了安全风险。需要沙盒（Sandbox）和权限控制机制，防止智能体的自我修改导致不可预见的后果。

Anthropic 内部如何使用 Claude

产品开发的实际工作流

Mike Krieger 透露了一个令人震惊的事实：Claude 的核心产品和 Claude Code 几乎完全由 Claude 自己编写完成。在 Anthropic，大多数产品基本可以说是 100% AI 编码。

这并不意味着人类开发者被取代了。相反，他们的角色发生了根本性转变——从"写代码的人"变成了"告诉 AI 写什么代码的人"。核心能力从"实现"转向了"判断"。

100% AI 编码

Anthropic 的产品开发流程已经成为一个教科书级别的 AI-native 实践案例。产品经理和工程师用自然语言描述意图，Claude Code 生成代码，团队审查和调整。整个过程中，AI 承担了绝大部分的编码工作。

一旦你知道要构建什么，实际的构建过程确实容易多了。 —— Mike Krieger

Skills 系统

Skills 系统是 Anthropic 内部的一项关键创新：将最佳实践打包成可复用的"技能"。例如 Mike 将 Dan 的"智能体原生"文章打包成 Skill，以后做原型开发时 Claude 就会自动以智能体原生的方式思考。

Skills 的创建流程：

  • 1
    发起请求
    "嘿 Claude，你能为此创建一个技能吗？"
  • 2
    元技能查找
    Claude 查找自己的"技能开发技能"（meta-skill），理解如何创建技能
  • 3
    创建与安装
    Claude 按标准流程创建新技能并安装到本地环境
  • 4
    重启生效
    重启 Claude Code 后，新技能即可在后续开发中自动生效

模板化与技能化的平衡是关键：好的模板和示例给模型提供良好的模式和范式，而 Skills 系统则让这些范式可以被系统性地复用和传播。

06

总结与启示

这场对话揭示了 AI 时代产品开发的六个核心观点：

  • 1
    效率不等于成功
    AI 能让你快速构建任何东西，但无法告诉你应该构建什么。产品判断力比开发速度更重要。
  • 2
    温室里的树不够强壮
    AI 生成的产品需要经历真实世界的磨练，才能长出坚实的根基。功能堆砌不等于产品价值。
  • 3
    重写的成本已经趋近于零
    AI 时代的重写不再是拖垮公司的一年计划，而是几天的实验。这彻底改变了产品迭代的节奏。
  • 4
    更早发布，更早验证
    不要假设在发布前就知道所有答案。Antfooding + 极简 V1 + 真实用户反馈，才是正确的循环。
  • 5
    智能体原生是未来方向
    软件需要从"预设路径"转向"灵活扩展"，让 AI 智能体能像用户一样使用所有功能。
  • 6
    Skills 系统是知识复用的关键
    将最佳实践打包成可复用的技能，让 AI 在每次开发中都能应用积累的智慧。
💡
AI 加速的是执行，而非判断
AI 时代最被低估的能力不是写代码的速度，而是知道什么该保留、什么该砍掉的判断力。正如 Mike Krieger 所说："一旦你知道要构建什么，实际的构建过程确实容易多了。"真正的产品能力，从来都不在于"能做多少"，而在于"该做多少"。
