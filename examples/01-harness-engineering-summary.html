01

前置概念回顾

Prompt Engineering（提示词工程）研究的是如何把发给大模型的话说得更清楚、更准确。比如给橘猫起名，从模糊的"帮我给猫起个名"，优化为"帮我给橘色小猫起名，两个字，体现活泼爱闹的性格"。如今因模型能力增强，这门技术已很少被单独提起。

Context Engineering（上下文工程）研究的是如何管理大模型接收到的所有信息——包括 Prompt、对话历史、工具列表等。核心方法包括上下文压缩（对过长的对话历史做总结以防溢出）。但 Context Engineering 的效果有上限，于是引出了我们今天的主角。

三者关系是层层递进

Prompt Engineering 研究怎么问问题 → Context Engineering 研究怎么给信息 → Harness Engineering 研究怎么搭建完整系统。研究范围不断向外扩展，关注的问题越来越大、越来越广。

02

什么是 Harness Engineering

Harness 的本意是"马具"——套在马身上用来控制马的缰绳、头套等装备。类比到 AI：大模型就像一匹强大的马，如果不加以控制就会像脱缰野马一样产生幻觉、偏离方向。Harness 就是用来控制大模型的系统。

Harness = Agent - Model
一个完整的 Agent 减去大模型，剩下的所有东西都是 Harness

以 Claude Code 为例：所有不属于 Claude 模型的部分都是 Harness——写在 CLAUDE.md 里面大模型要遵循的规则、Claude Code 可以使用的工具、定时调度机制等。

顺理成章地，Harness Engineering 就是一门专门研究如何构建与设计 Harness 的技术——除了大模型本身不研究，别的什么都研究。它不再是仅仅盯着模型输入的那点提示词或上下文，而是站在更高的系统层面上，研究怎么给大模型设计一套可以稳定运行的系统。

维度 Prompt Engineering Context Engineering Harness Engineering
研究焦点 怎么问问题 怎么给信息 怎么搭建系统
作用范围 单次交互 单会话 系统级、全生命周期
一句话总结 教 AI 听懂一句话 给 AI 递对资料 给 AI 建一座工厂
03

OpenAI 的 Harness 实战

2025 年 8 月，OpenAI 内部启动了一个疯狂实验：用 AI 从零写一个真实的软件产品，全程不允许工程师手写一行代码。产品所有组成部分——逻辑、测试、CI配置、文档、内部工具——全部由 AI 生成。

实验成果

5 个月，3→7 人团队，近 100 万行代码，线上真实用户的生产系统。开发效率约为纯人工的十倍。

但实验一开始并不顺利——不是因为模型不够聪明，而是 Harness 没有搭好。Agent 经常走偏方向、重复犯错。于是他们做了大量优化，可分为三大类：

①
上下文管理

最初把所有规范塞进一个超大 AGENTS.md 文件，但发现两个问题：

✕ 大而全的方案

内容太多导致模型效果变差，就像新人报到被砸了一本巨厚员工手册，完全不知道重点在哪。

文件逐步腐化，时间一长变成一堆过时的垃圾，连人都懒得整理。

✓ 改进后的方案

AGENTS.md 压缩到约 100 行（只做索引），相关文档放在对应目录下，按需加载。

强制要求所有重要决策搬进代码仓库，让仓库成为唯一的事实来源。

②
验证与反馈
  • 1
    UI 自动验证

    给 Codex 配上 Chrome DevTools，使其可以自己截图、查看 DOM 结构、模拟用户操作来验证 UI 是否符合要求。

  • 2
    可观测性工具链

    接入完整的日志、指标和运行链路追踪，每个任务跑在完全隔离的环境里。

  • 3
    架构规范自动闭环

    使用 Linter 和测试保证分层架构（Types → Config → Repo → Service → Runtime → UI，严格单向依赖）。Agent 写代码 → Linter 检测 → 报错反馈 → Agent 修改 → 再检测，直到全部通过。

③
技术债清理

设置后台 Codex 任务定期扫描代码库，找出偏离规范的地方自动修改提交；同样设置后台任务定期扫描文档库，找出过时文档自动修复。无论是代码还是文档，OpenAI 都有对应的维护方案，不会放任自流。

"Human Steer, Agents Execute"——人类负责掌舵（定方向、给上下文、制定规则），Agent 负责干活。软件工程师的核心职责从写代码，变成了为 Agent 搭建稳定可靠的系统与支撑框架。

—— OpenAI 官方文章核心理念
04

Anthropic 的 Harness 实战

Anthropic 发表了两篇核心文章，总结下来最关键的两点是：任务规划和质量评估。

① 任务规划——从失败中学习

让 Agent 直接克隆 Claude.ai（一个功能丰富的聊天界面），Agent 急于求成，一口气想做完全部功能，结果干到一半上下文满了就丢下烂摊子。下一个 Agent 接手时完全不知道前面发生了什么，只能靠猜，有些功能做了一半却被宣布完工。

解法：引入 Initializer Agent 拆解用户需求为详细功能列表，干活 Agent 逐个功能点稳扎稳打、做完一个标记一个。第二篇文章进一步优化，把需求拆解单独做成 Planner Agent。

② 质量评估——不能自己评自己

✕ 方案一：人工评估

效率太低，都 AI 时代了，能交给 AI 的都交给 AI。

✕ 方案二：Agent 自评

"王婆卖瓜自卖自夸"，Agent 对自己做的产品天然有滤镜，即使有明显的 Bug 也会视而不见、给自己打高分。

最终方案：独立 Evaluator Agent

做一个专门的质量评估 Agent，作为独立第三方来评估产出质量。把生成代码和质量评估拆开，分别交给两个不同的 Agent。

经典三 Agent 架构（Full Harness 方案）详细实现

Anthropic 把这个包含三个 Agent 的方案叫做 Full Harness 方案。相比之下，只靠一个 Generator 独立完成所有需求的传统单 Agent 模式被称为 Solo 方案。下面是 Full Harness 方案的完整实现流程。

阶段一
Planner — 需求拆解（仅执行一次）

用户提出一句模糊的需求（如"帮我做一个游戏制作工具"），Planner Agent 负责把这句模糊的需求扩展成一份完整、清晰的功能列表。

关键设计

在第一篇文章中，这个角色叫 Initializer Agent（负责初始化环境、拆解需求、编写启动脚本、添加进度文件）。第二篇文章把其中最核心的"拆解需求"单独拎出来，做成了独立的 Planner Agent。后续 Generator 拿着功能列表直接干活就行，不用再对着用户需求猜了。

阶段二
Generator + Evaluator — 交互标准协商

Generator 从功能列表中挑选一个功能点，但不会立刻动手写代码。它先跟 Evaluator 讨论这个功能点的交互标准——到底做到什么程度才算完成。

1
Generator 提出方案：把对这个功能点的实现想法发给 Evaluator
2
Evaluator 提出修改意见：可能对提议有补充或纠正
3
Generator 根据意见修改：再次提交新的交互标准
⟳
循环直到 Evaluator 确认：重复以上过程，直到 Evaluator 认为标准没有问题为止
阶段三
Generator — 代码实现

交互标准确认好之后，Generator 开始编写代码来实现这个功能点。实现完毕后，Generator 把实现结果提交给 Evaluator。

阶段四
Evaluator — 质量评估与修复循环

Evaluator 对 Generator 的实现结果做出评估反馈。这是整个方案中最重要的质量关卡。

1
Evaluator 评估：独立审查代码实现，判断是否达到交互标准
2
如果不通过：把问题列表反馈给 Generator
3
Generator 修改代码：根据反馈做相应修改
4
再次提交评估：修改后重新提交给 Evaluator
⟳
循环直到通过：重复直到 Evaluator 评估通过为止
为什么不能自评？

Agent 自评就是"王婆卖瓜自卖自夸"，天然有滤镜，即使有明显的 Bug 也能视而不见给自己打高分。独立 Evaluator 作为第三方没有理由去袒护别人的产出，评估结果自然客观得多。而且把 Evaluator 单独拎出来，还可以单独优化和训练它的评估能力。

阶段五
检查功能列表 — 循环往复，逐个击破

一个功能点评估通过后，并不意味着结束。Generator 需要回到功能列表，检查是否还有未完成的功能点。整个阶段二到阶段四会循环往复，直到所有功能点全部开发完毕。

✓
功能点 N 评估通过 — Evaluator 确认当前功能点质量达标
↓
标记完成 — 在功能列表中将该功能点标记为已完成
?
检查功能列表 — 功能列表中是否还有未完成的功能点？
→
是 → 回到阶段二 — Generator 从列表中挑选下一个功能点，重新走协商标准 → 写代码 → 评估修复的完整流程
✅
否 → 全部完成 — 所有功能点均已通过评估，可以交付最终产品

Anthropic 在提示词里强制 Generator 每次只执行一个功能点，做完一个标记一个再做下一个，否则 Agent 会急于求成一口气做所有功能，结果上下文满了就丢下烂摊子。

模型升级后的简化

在升级到 Opus 4.6 后，Generator 的全局统筹能力够强了，可以一次把所有功能点都拿过来自己决定先做哪个再做哪个，稳步推进，不再需要别人对执行流程指指点点。Evaluator 也直接评估最终产出即可，不需要再分功能点评估。Anthropic 后面就把"逐个功能点执行"的强制约束去掉了，简化为 Generator 自主统筹 + Evaluator 最终把关的流程。

05

概念起源与传播

Harness 这个词并不算新词。传统软件测试领域就有 Test Harness（支持测试代码运行的框架），AI 领域也有 LM Evaluation Harness。大家一直在默默用，谁也没觉得需要大吹特吹。

重点是 Harness Engineering 这个组合词。目前比较公认的传播时间线如下：

2026-02-05
Mitchell Hashimoto（HashiCorp 联合创始人）发表文章 "My AI Adoption Journey"，首次提出 Harness Engineering 一词。核心理念：Agent 犯了错就去改造系统，让它绝不再犯同样的错。原文说"也不知道业界有没有公认的叫法，我就姑且叫它 Harness Engineering，有更好的词随时改口"。
2026-02-11
OpenAI 发表 Harness Engineering 文章，信息量极大，迅速在业界引发巨大反响。真正引爆了这个概念。
2026-02-17
Martin Fowler 网站发文 "Harness Engineering: First Thoughts"，作者指出 OpenAI 文章正文其实只用了一次 Harness 这个词，推测可能是受到 Mitchell Hashimoto 启发后临时放到标题里。
2026-03-10
LangChain 发文 "The Anatomy of the Agent's Harness"，第一次明确给出公式：Agent = Model + Harness。公式一出，概念就算定下来了。
2026-03-24
Anthropic 发文，拿出 Planner + Generator + Evaluator 经典架构。虽然自己只用 Harness 名词、没有套用 Harness Engineering 这个新词，但整个 AI 圈心照不宣，直接把这套架构当成了 Harness Engineering 的教科书级案例。
微妙的事实

Harness Engineering 里面用到的所有技术——Linter 代码检查、任务拆解、质量评估机制——竟然没有一个是从未有过的新技术。它提供的是一套新的系统思维架构，而不是发明了一批颠覆性的新技术。

06

是噱头还是突破？

反对者的攻击点主要有两个：

  • 1
    没有新技术，全是新瓶装旧酒

    Linter、任务拆解、质量评估早就有了，特意造个新词到处宣传，这不就是噱头吗？

  • 2
    所有 Harness 迟早会被淘汰

    随着模型能力增强，今天的 Harness 设计可能会被模型自身逐步吸收，最终变得不再需要。

而且这种担忧连 Anthropic 自己的文章里都有迹象——从 Sonnet 4.5 升级到 Opus 4.6 后，上下文焦虑被大幅缓解，不再需要上下文重置；强制逐个功能点执行的机制也不再需要了。

🎯

UP 主观点：不是噱头，但也不是终局

不是噱头——OpenAI 和 Anthropic 都通过它把 Agent 的稳定性、自动化程度往前推了一大步。工程领域的真正进步往往不在于发明新技术，而在于有统一的框架把零散能力组织起来。

不是终局——随着模型能力继续增强，今天用来约束、纠正模型的系统设计可能会被模型自身逐步吸收。未来可能只需配置最基础的 Harness，模型就能自己搞定 99% 的问题。

Harness Engineering 是一个过渡期的关键技术。它可能不是未来的终局答案，但它是当下最现实的答案。在模型依然会犯错、产生幻觉、偏离轨道的今天，谁能把 Harness 搭得更稳，谁就能更早地把 AI 能力转化为真正的生产力。

—— 马克的技术工作坊

OpenAI Engineering

工程技术：在智能体优先的世界中利用 Codex

作者：Ryan Lopopolo · 2026 年 2 月 11 日
核心思想

OpenAI 用 5 个月时间，让 3 名工程师借助 Codex 智能体构建了一款约 100 万行代码的产品，并且没有一行代码是人工编写的。人类不再写代码，而是专注于设计环境、明确意图、构建反馈回路。

简要讲解

1
人类掌舵，智能体执行。工程师的工作重心从写代码转向了系统设计、架构决策和杠杆效应。当 Agent 遇到困难时，解决方案不是"再努力一点"，而是追问"还需要给它什么能力和约束"。
2
给地图，别给说明书。OpenAI 最初尝试写一个巨大的 AGENTS.md，结果发现信息太多反而有害——Agent 无法判断哪些信息仍然有效，哪些已经过时。最终方案是：AGENTS.md 只做"目录"，约 100 行，真正的知识库放在结构化的 docs/ 目录中，实现渐进式披露。
3
强制执行不变量，而非微观管理。每个业务域严格分层（Types → Config → Repo → Service → Runtime → UI），依赖方向通过自定义 linter 机械地强制执行。Agent 在边界内拥有很大自由度，但越界行为会被立即拦截。
4
技术债就像高息贷款。完全自主的 Agent 会复制代码库中已有的模式，包括不理想的模式。OpenAI 建立了"黄金原则"和循环清理流程，定期让 Agent 扫描偏差并发起重构 PR——类似垃圾回收机制，小额持续偿还远胜于积压后一次性处理。
5
让 Agent 能直接看到应用。为了让 Codex 做 QA，OpenAI 接入了 Chrome DevTools 协议，让 Agent 能截图、操作 DOM、复现 Bug、验证修复。单次运行可持续工作超过 6 小时。
Anthropic Engineering

Effective Harnesses for Long-Running Agents

作者：Justin Young · 2025 年 11 月 26 日
核心思想

长时间运行的 Agent 面临的核心挑战是：每次新会话都从零开始，没有上一次的记忆。就像轮班制的工程师，每一班新来的人都不知道上一班做了什么。Anthropic 提出了Initializer Agent + Coding Agent 的两阶段方案来解决这个问题。

简要讲解

1
Agent 的两种失败模式。第一种：试图一次性完成所有工作（one-shot），结果上下文耗尽，留下一半实现的烂摊子。第二种：做了几个功能后，环顾四周觉得"差不多了"，提前宣布完工。
2
Initializer Agent 做初始化。首次运行时，专门初始化环境：创建功能列表文件（feature_list.json，含 200+ 个功能点，全部标记为"failing"）、进度文件（claude-progress.txt）、启动脚本（init.sh）和初始 git commit。为后续所有 Coding Agent 提供清晰的工作地图。
3
Coding Agent 逐个功能推进。每次只做一个功能点。会话开始时先运行 init.sh 启动开发服务器，用 Puppeteer 做基本验证确保没有基础 Bug，然后从功能列表中挑选最高优先级的未完成功能来实现。做完后写 git commit 和进度更新。
4
用 JSON 而非 Markdown 存储功能列表。实验发现 Agent 更容易随意修改或覆盖 Markdown 文件，而 JSON 格式更结构化，模型更不容易误操作。配合"严禁删除或编辑测试"的强措辞指令，确保功能列表不被篡改。
5
必须用浏览器做端到端测试。Claude 倾向于用单元测试或 curl 命令验证功能，但经常忽略端到端的实际体验。给它 Puppeteer MCP 让它像人类一样操作浏览器后，性能显著提升——能发现从代码层面看不出来的 Bug。
Anthropic Engineering

Harness Design for Long-Running Application Development

作者：Prithvi Rajasekaran · 2026 年 3 月 24 日
核心思想

受 GAN（生成对抗网络）启发，Anthropic 设计了Planner + Generator + Evaluator 三 Agent 架构。关键创新：让评估者独立于生成者，解决了 Agent 自评时"王婆卖瓜"的问题。这也是视频重点讲解的 Full Harness 方案。

简要讲解

1
为什么自评不行？Agent 给自己打分时，会不自觉地给出高分——即使人类一眼就能看出质量平庸。尤其在主观任务（如设计）上，不存在二元的"通过/不通过"判定，Agent 倾向于对 LLM 生成的作品格外宽容。解决方案：把"干活的人"和"验收的人"分开。
2
把主观审美变成可评分的维度。Anthropic 定义了四个评估维度：设计质量（整体感）、原创性（非模板化）、工艺（排版/间距/对比度）、功能性（可用性）。重点加权重在前两个维度，因为后两个 Agent 本身就能做好。
3
Sprint Contract 机制。在实现每个功能点之前，Generator 和 Evaluator 先协商"做完长什么样"——Generator 提出方案，Evaluator 审查方案，双方来回迭代直到达成一致。这避免了 Generator 理解偏差导致做出来的东西南辕北辙。
4
上下文重置 vs 压缩。Sonnet 4.5 存在"上下文焦虑"——接近上下文窗口限制时会草草收工。解决方案是完全清空上下文重新开始（重置），而非只压缩历史记录。升级到 Opus 4.5 后这个问题大幅缓解，可以不用重置直接持续工作。
5
20 分钟 vs 6 小时的天壤之别。Solo 方案（单 Agent）20 分钟完成、花费 $9，但输出基本没法用。Full Harness 方案 6 小时完成、花费 $200，输出达到可用水准。这和人类一样——考 60 分可能只需 3 天，但考 90 分可能需要一个月。
