1

传统循环的困境

回合制编码循环

大多数 AI 编码会话都遵循一个固定的循环模式：你提出修改需求，Agent 收集上下文、执行操作、返回结果。你检查结果，提出下一轮修改——如此往复。这就像一场回合制游戏，每一步都在等待你的输入。

有时候你需要来回沟通好几轮才能得到满意的结果。因为就像人类一样，Agent 在收到反馈时才能产出更好的工作成果。但问题在于，这种反馈不一定每次都必须来自你本人。

人工反馈瓶颈

传统模式

你提出修改 → Agent 执行 → 你手动检查 → 你提出反馈 → Agent 修改

每一步都依赖人工介入，形成反馈瓶颈。你需要不断"看孩子"。

自动验证模式

你提出修改 → Agent 执行 → Agent 自我验证 → Agent 修改 → 返回最终结果

Agent 利用工具链自行检查，减少来回沟通，释放你的时间。

核心观点

反馈不必总是来自人类。随着模型在编码和推理方面的能力提升，它们也越来越擅长通过观察周围的信号来验证自己的工作。

2

Agent 的自我验证能力

利用环境信号

Claude 可以从代码库中的多种信号来验证自己的工作：类型检查器（Type Checker）、单元测试（Unit Tests）、运行时错误（Runtime Errors）等。通常，只要这些工具存在于你的代码库中，Claude 就能理解应该运行哪些命令以及如何修复代码。

Type Checker Unit Tests Runtime Errors Linter Build Errors

确定性检查的局限

然而，这些检查都是确定性的——要么通过，要么失败；代码要么能运行，要么不能。Claude 无法总是推断出的是你手动验证一个新功能或修改时所采取的步骤，因为这些步骤通常并不存在于代码库中。

你为 Claude 编码的验证步骤越多，它的第一次响应就越接近你心目中的理想结果。你将花更少的时间"看孩子"，减少来回沟通的 Token 消耗，腾出精力去做其他事情。

各平台验证工具

无论你的手动验证是什么样的，通常都有办法把它交给 Claude 来做。以下是不同平台的对应方案：

  • FE
    前端 (Web UI)
    启动开发服务器、打开浏览器、以用户身份点击交互、检查控制台错误。Claude 可使用 Chrome Extension、Desktop Preview 和 Chrome DevTools MCP 来打开页面、导航、截图、读取控制台并检测 Core Web Vitals。
    Chrome Extension Desktop Preview Chrome DevTools MCP
  • BE
    后端 (API/Service)
    调用端点（Endpoint）、检查状态码和响应内容。Claude 可以直接发起请求并分析返回结果。
  • MB
    移动端/桌面端
    通过 Computer Use 以及 iOS / Android 模拟器 MCP 来驱动应用界面验证。
    Computer Use iOS Simulator MCP Android Simulator MCP

创建验证 Skill

用 Skill Creator 编码

一旦你理清了当前的验证工作流，就可以通过调用 Skill Creator 将其编码为一个 Skill。你可以要求 Claude 以访谈的方式引导你完成流程描述。

访谈式创建

  • 1
    调用 Skill Creator
    在 Claude Code 中调用 Skill Creator，并要求 Claude 通过访谈帮你梳理验证流程。
  • 2
    描述领域最佳实践
    如果难以表述，可以请 Claude 帮助你梳理领域最佳实践，并展示一个端到端验证流程的样子。
  • 3
    审查并保存 Skill
    Claude 会生成一个包含完整验证步骤的 Skill 草稿，你可以审查并根据需要调整。
Skill 草稿示例

视频中的演示 Skill 包含了以下验证环节：通过浏览器自动化验证变更、捕获视觉证据（截图）、监控控制台输出、检测功能回归问题。这意味着每次 Claude 完成 UI 修改后，都会自动运行这一整套验证流程。

实战演示：自动捕获 Layout Shift

Like 按钮案例

视频中展示了一个实际案例。作者有一个页面，他知道页面上存在布局偏移（Layout Shift）问题，但一直没有去修复。他让 Claude 做一件完全无关的事情——添加一个 Like 按钮。

Claude 添加了按钮，然后因为这是一个 UI 变更，自动运行了验证流程：启动开发服务器、打开预览、点击 Like 按钮确认功能正常、截图记录。

自动发现问题

关键在于，Skill 还指示 Claude 通过 Chrome DevTools MCP 捕获性能追踪数据。于是 Claude 报告回来：

Like 按钮功能正常，但页面上存在布局偏移（Layout Shift）。这并不属于本次任务的范围，但因为检查 Core Web Vitals 是我们创建的验证工作流的一部分，所以 Skill 自动捕获了这个问题。

-- Claude 的验证报告
洞察

这是一个小例子，但你可以想象，随着你编码的工作流越来越多，Claude 的覆盖范围也会不断增长。那些你"一直想修但没时间修"的问题，会在每次相关变更时被自动发现。

第二双眼睛：Agent 代码审查

独立 Agent 的优势

在将代码推送到生产环境之前，值得让另一个 Agent 来审查代码。第二个 Agent 不会有编写代码的那个 Agent 相同的偏见——它拥有独立的上下文，不会受到之前对话的影响。这种隔离性使得审查更加客观，能够发现第一个 Agent 可能遗漏的问题。

审查方案选择

  • A
    Code Review Plugin
    并行启动多个子 Agent，每个子 Agent 从不同角度审查代码变更，对发现的问题进行置信度评分，并将结果发布到 PR 上。
  • B
    Managed Claude Code Review
    面向 Team 和 Enterprise 计划，通过 GitHub 在每个 PR 上自动运行代码审查，无需额外配置。
关键建议

无论你选择哪种方案，在合并之前拥有一道最后防线是非常有价值的。这让你的代码质量保障形成两层：构建时的验证层和合并前的独立审查层。

端到端工作流编排

Skill 串联

到目前为止，你已经有了两层保障：Claude 构建应用时运行的验证层，以及合并前由独立 Agent 执行的审查层。这两层都属于同一个更大的开发周期的一部分。

你可以把这些步骤编排成一个工作流——编写一个调用其他 Skill 的 Skill。例如，Claude Code 团队自己就有一个在发布功能时运行的 Skill：

完整发布流程

1
Build & Simplify
清理 diff，精简代码变更
→
2
Custom Verify
端到端验证变更是否正常工作
→
3
Design Check
如果 diff 涉及 UI，检查设计一致性
→
4
Open PR
创建 PR 并订阅通知
→
5
Watch CI
监控 CI 并自动修复失败
★

核心思想

你的工作流可能看起来不同，但核心原则是一样的：尽可能多地将验证步骤交给 Claude，让它能够端到端地自我验证工作成果。你的角色从"每一步都盯着"转变为"设计好验证流程，让 Agent 自主迭代"。
